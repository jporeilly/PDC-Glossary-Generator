Build a Business Glossary for Pentaho Data Catalog

Scan your data sources, review and refine suggested terms, assign stewardship, and export import-ready JSONL.

The workflow

1
Connect
Add a connection for each source — a database, a MinIO/S3 document store, or a DDL file. Scan one to start; Add to glossary from others to span structured and unstructured data. The Connections page profiles what it finds.
2
Review & prune
Every column becomes one candidate term, so you prune rather than hunt for gaps. Edit definition, purpose, sensitivity and tags inline; filter and use Keep High+Med conf to cut noise. You can also open an existing glossary export for review, or enhance your scan against one.
3
Govern & generate
First approve any scan-grown pending terms/tags on the Dictionary page — only governed vocabulary flows into the Registry. Then manage the user roster (add manually or fetch live from Keycloak) and set steward, owner, custodian, status, rating and reviewed-date — globally or per category. Then Generate JSONL right here, with stewardship baked in, and import via PDC's Business Glossary → Import.
4
Resolve Term IDs & apply
Once the glossary is imported, push the term↔column links over the API: authenticate once, resolve IDs, then apply with a dry-run before any write. Sensitivity, CDE, verified-lineage and trust score follow.
The full working cycle — exact order, and why it matters

One complete pass, from scan to a Registry the Policy Generator can consume. The order matters because each step feeds the next:

  1. Scan or resume. Connect & scan a source, or let the app auto-resume your last saved glossary.
  2. Review the grid. The AI agents assist — Enrich, AI suggest (evidence), AI QA, AI categorize (fills only blank/generic categories) — and the duplicate advisor recommends Merge / Disambiguate / Keep separate. Rename divergent names to their canonical term; the vocabulary’s aliases fold them automatically on future scans.
  3. Dictionary: review pending. AI review advises per candidate; the alias action folds near-duplicates. Approve only what belongs — approved items govern the Registry and export into the pack. Mistakes are reversible per item (✕ retire / ⤵ fold), and a retire is durable: tombstoned through reseeds, offered for removal from the pack at export.
  4. Suggest tags (grid) after any vocabulary change — re-derives every row’s tags from the governed allow-list and accretes usage, which fills the facet preview. (Freshly reseeded counters are all zero — that means “no scan yet”, never “retire everything”.)
  5. Govern. Roster-driven steward/owner/custodian, ratings, review dates.
  6. Save glossary → Generate. Generate writes the import JSONL and the Registry. In PDC: Business Glossary → Import (if terms were renamed, delete the old glossary first — ids are name-based, so renames mint new terms). Then Resolve & stamp IDs (backfills real ids into the Registry) and Apply to PDC.
  7. Export domain pack → Apply to this app → commit. Last, because it exports the reviewed state of everything above. Decide any conflict rows (curation keeps the pack’s value by default; seeds prefer fresh scan evidence; steward-retired entries default to removal). Apply writes the pack and reseeds the dictionary in one click — you never run Reseed separately unless you hand-edit the pack file. Commit the pack to the scenario repo so every future install starts from it.

After step 6 the Registry (registries/registry.<glossary>.json) is current; after step 7 the flywheel is closed. Save dictionary is only needed after hand-editing tags/rules — approvals and scan accretion persist on their own. App state survives git pull untouched; Settings → State snapshot covers machine moves and restore points.

Best practices

  • Terms export as Draft — proposals until a Business Steward approves them. Review before importing.
  • The import replaces the whole glossary; reuse _ids (open an export for review) to update in place.
  • Confidence is a review signal: High = a DB comment or key; Low = templated from the name. Enhance against an existing glossary or add column comments to raise it.
  • CDE (Critical Data Element) is auto-inferred from keys, sensitivity, and critical/compliance terms — and is always the steward's to confirm.
  • UUIDs are per-instance (Keycloak). Fetch the roster from the target instance so bindings resolve.

Connections

Each data source is its own saved connection. Scan one to start a glossary, then Add to glossary from others to span structured and unstructured sources.

Connection types & what each button does
Database (live scan)Connects to PostgreSQL/MySQL/SQL Server with a least-privilege read-only user and introspects schema, keys and comments. Sample values additionally reads a small sample to refine sensitivity and Data Quality. Object store (MinIO/S3)Browses a bucket over the S3 API and treats each file as a document term. Use the host/VM IP for the endpoint, not localhost (MinIO needs path-style addressing). DDL fileParses a CREATE TABLE script when you can’t reach the live database — same suggestions, no connection needed. ScanReads the source and starts a fresh glossary from it (replaces the current candidate terms). Add to glossaryScans another source and merges its terms into the existing glossary — this is how you span a database and a document store in one glossary. DiscoverRuns deeper profiling (distribution, uniqueness, patterns) so confidence and Data Quality are evidence-based rather than name-based. TestValidates the connection details before saving — the source stays offline until it passes (just like PDC).

Nothing is written to the source — every call is read-only. The exact SQL/S3 it runs is in the panel at the bottom of this page.

Bulk-load data sources into PDC setup step — runs before the glossary

Register many sources in PDC at once from a CSV. For each row the app creates the data source, then triggers a metadata ingest scoped to it and waits for the job — the Connect + Ingest steps that precede profiling, identification and the glossary. Use kind = postgres, mysql, oracle, minio/s3, or azure_blob. Download a sample CSV, or Export existing to turn the connections you saved by hand (the cards below) into a loader-ready CSV. Secrets are sent to PDC only and never saved by the app; an export of your saved connections does include their credentials, so treat that file as sensitive. A connection that already exists in PDC is detected and shown as EXISTS — it’s re-scanned, not re-created, so re-runs don’t hit a 400 duplicate error.

Two things that make an ingest report OK but find nothing — or fail: (1) set schemaNames to the schema your tables actually live in (e.g. cscu_core, not public) — an ingest into an empty schema completes green but harvests zero terms. (2) Object stores (minio/s3) need container (the bucket), a reachable endpoint, and the bucket to actually contain files; metadata ingest lists the files, and content-level classification then needs PDC’s Data Discovery step. Ingest failures now show PDC’s reason in the note column. (3) To scope a scan, add includePatterns / excludePatterns columns (semicolon-separated globs, e.g. excludePatterns=*.md;*.tmp) — they flow into both the create and the scan job.

Glossary

    
Review suggested terms — definition, purpose, sensitivity, CDE and tags — together, and prune. Tags are drawn from the governed vocabulary on the Dictionary page; stewardship and JSONL generation come next, on the Govern page.


    
How terms are defined & built
      
Every column the scan finds becomes one candidate term — so you prune rather than hunt for gaps. All seven fields below are derived locally by suggester.py at scan time (the SQL it reads is shown on the Connections panel); there are no extra API calls to build them. The one optional call from this page is LLM enrichment. Edit any field inline — your edit always wins over the suggestion.

      

        TermThe column name made human-readable: service_address → “Service Address”.
        DefinitionThe database column’s COMMENT if it has one; otherwise “Unique identifier…” for a primary key, a “Reference linking…” line for a foreign key, or a name-based template. Enrich with LLM can rewrite any of these.
        PurposeA business “how it’s used” sentence: identity/lineage wording for keys, privacy wording for PII (name, contact, address, financial), otherwise a per-category template (Customer, Billing, Usage, Governance…).
        SensitivityPattern rules on the name: email / SSN / account number / address → HIGH; phone / personal name / city · zip → MEDIUM; everything else LOW. With Sample values on, what’s actually in the data overrides the name-based guess.
        CDEFlagged True when the column is a primary key, is HIGH sensitivity, is financial / personal-name PII, or matches a critical pattern (account number, meter id, balance, compliance, lead level, pH, turbidity…).
        TagsThe category, plus PII / privacy / financial labels, plus maskable (HIGH), CDE and identifier (keys) — capped at six.
        ConfidenceAn evidence signal, not a quality score: High = a DB comment, a key, or a profiling hit in the data; Medium = the name matched a PII pattern; Low = templated from the column name alone.

Resolve Term IDs

    
Push the glossary you built to your PDC instance over the public API — authenticate once, resolve term IDs, then apply with a dry-run. The glossary itself is imported in PDC’s UI; everything here writes the term↔column links, sensitivity, CDE, verified-lineage and rating onto the columns.


    
Why generate & import the glossary before you resolve?
      
A business-term link only binds to its glossary when it carries both an id and a glossaryId — and those identifiers don’t exist until the term itself exists inside PDC. The term is created when you import the generated JSONL (PDC’s Business Glossary → Import). So the order is forced by PDC’s data model:

      
Govern → Generate JSONL → Import in PDC → Resolve → Apply.  Resolve searches PDC for each term by name and stamps the real id/glossaryId back onto your links; Apply then PATCHes those resolved links (plus sensitivity, CDE, lineage and rating) onto the column entities. Skip the import and Resolve finds nothing — the columns would get a name-only term with no glossary behind it (the “—” you see in PDC).

How stewardship & auto-assign work
      
Governance is baked into the JSONL at generate time, so set it here before you generate. Every term carries three roles, and you can set them once as defaults, then override per category.

      

        Business StewardOwns the meaning of the term — accepts the definition and is accountable for it in PDC. Mapped from the Keycloak Business_Steward role.
        OwnerOwns the governance of the data behind the term. Mapped from Data_Steward.
        CustodianThe technical keeper of where the data is stored. Mapped from Data_Storage_Administrator.
        UUID bindingA person only resolves in PDC by their Keycloak UUID, which is per-instance — so a binding made against one instance won’t resolve on another. Fetch the roster live to guarantee the ids match.
        Auto-assignFor each category it scores every person on (1) expertise keywords vs. that category’s terms/columns, then (2) their role. Best expertise match wins; with no match it falls back to the role default. Each card shows the rationale for who was picked. Manual edits lock that field so a re-run won’t overwrite you.
        RatingA 1–5 quality star, set globally or per category. Auto (scan DQ) derives each category’s rating from its columns’ scan Data-Quality scores.

Under the hood — the pack flywheel: whose scan feeds the pack?
      
Both scans play a role, at different stages — but the pack only ever receives your reviewed state, never raw scan output:

      
    
        
  1. PDC's scans supply the raw evidence. Data Profiling and Data Identification run inside PDC and produce the profiled sample values, statistics and identification tags on each entity.
    2. 
        
  2. This app's scan interprets it. Suggest terms harvests those entities over the v3 API and induces structure on top: value patterns and signatures from PDC's profiled samples, enum lists, PK/FK keys, plus proposed terms, tags, categories and sensitivity. (The duplicate advisor's live value probe is the one exception — it samples the source database directly.)
    3. 
        
  3. Your review is the gate. Keep/reject rows, run the AI agents, approve terms and tags into the governed dictionary. Nothing pending or rejected goes further.
    4. 
        
  4. Export domain pack reads only the reviewed layer. Kept rows drive the table mappings, learned abbreviations and curated_seeds; the approved company vocabulary drives the terms, tags and rules. It merges over the installed pack — additions fill gaps, and any disagreement between scan and pack is listed for you to decide (curated seeds default to the fresher scan evidence; everything else keeps the pack's value until you tick otherwise).
    5. 
      

      
No pack yet? Run packless — the built-in generic defaults still route obvious columns. After one scan → review cycle, the first export is your base pack: commit it to the scenario's domain_pack/ folder and every later cycle refines it. Apply to this app installs the refreshed pack here (timestamped backup, approved items survive the reseed); committing it makes the improvement permanent for every future install.

      
When in the cycle: Export domain pack is the last step of a pass — after grid review, pending approvals, Suggest tags, and Save glossary → Generate → import → Resolve → Apply — because it exports the reviewed state of all of it. The Home page’s “full working cycle” panel lays out the exact order and the why.
